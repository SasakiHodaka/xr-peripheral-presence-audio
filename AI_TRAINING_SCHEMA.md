# AI Training Schema

## Selected Method

The first AI target follows this practical pipeline:

```text
Unity situation simulation
-> candidate cue evaluation
-> evaluated cue-label dataset
-> cue-control model
-> PeripheralCueModel / PeripheralCueAudioEmitter
```

NAF remains a background reference. The first trainable model should predict compact cue-control parameters, not full impulse responses and not an acoustic field.

The current developer-selected cue labels are only an initial prototype baseline. They are useful for checking whether the Unity and training pipeline run, but they are not reliable final ground truth. Final labels should be built from simulation and evaluation.

## Label Reliability Problem

Simulation-based audio work such as Meta Audio Simulator, SoundSpaces, SoundSpaces 2.0, Neural Acoustic Fields, and self-supervised acoustic learning can generate many samples without asking humans to label every sample.

However, this project has a different target:

```text
SoundSpaces-style input:
room geometry + materials + source position + listener position

SoundSpaces-style target:
simulated RIR or acoustic response
```

That target can be generated by physics simulation.

This project's target is:

```text
input:
distance + direction + approach speed + speaking + crossing + view state

target:
footstep / voice / ambient presence / none
presenceScore
volumeGain
```

This target cannot be decided by physics alone because the correct cue depends on human awareness, clarity, naturalness, and discomfort. Therefore, the final dataset must use evaluation-derived labels.

## Target Columns

The prototype model predicts:

- `cueType`
- `presenceScore`
- `volumeGain`
- `cueLowPassHz`
- `cueReverbAmount`
- `cueOcclusionGain`

These are compact, Unity-facing targets.

For the final cue-label dataset, the targets should be:

- `cueType`: the best-performing cue candidate for the generated situation
- `presenceScore`: a score derived from perceptual/evaluation results and situation urgency
- `volumeGain`: a playback gain derived from `presenceScore`, distance, and discomfort constraints

Candidate presence score formula:

```text
presenceScore =
  a * localizationScore
  + b * reactionTimeScore
  + c * approachRecognitionScore
  + d * clarityScore
  - e * discomfortScore
```

The first version can use fixed transparent weights. Later versions can optimize the weights from study data.

## First Model Baselines

Start with small supervised baselines:

- classifier for `cueType`
- regressor for `presenceScore`
- regressor for `volumeGain`
- regressor for `cueLowPassHz`
- regressor for `cueReverbAmount`
- regressor for `cueOcclusionGain`

Recommended order:

1. RandomForest or Gradient Boosting for fast tabular baselines.
2. Small MLP or NN after the evaluated label schema stabilizes.
3. Audio-visual or environment-acoustic model only after cue-label generation is reliable.

Compare at least three label/model sources:

1. rule-based cue model
2. developer-selected prototype labels
3. evaluation-derived labels

## Current Training Script

Generate a reproducible simulation-first dataset without opening Unity:

```powershell
python Tools/generate_simulation_dataset.py --mode grid --output cue_training_dataset.csv
```

For a broader randomized dataset:

```powershell
python Tools/generate_simulation_dataset.py --mode random --random-count 5000 --output cue_training_dataset_random.csv
```

The current dependency-free baseline is:

```powershell
python Tools/train_cue_model.py --dataset cue_training_dataset.csv --classifier linear --classifier-epochs 220 --epochs 80
```

It trains a lightweight cue-control model from `cue_training_dataset.csv` and writes:

- `Assets/Models/cue_model_unity.json`
- `cue_training_predictions.csv`

`Assets/Models/cue_model_unity.json` is a Unity-friendly JSON file. Assign it to
`PeripheralCueModel.learnedModelJson`, then set
`PeripheralCueModel.comparisonCondition` to `LearnedCue` to use the trained
model in Play Mode. If the asset is not assigned, `PeripheralCueModel` falls
back to the existing rule-based cue prediction.

The current default classifier is a dependency-free linear multi-class classifier for `cueType`, plus linear regressors for the numeric cue parameters. The older centroid classifier remains available:

```powershell
python Tools/train_cue_model.py --classifier centroid
```

Recent simulation check:

- grid dataset: 800 rows, `cueType` test accuracy 0.955
- random dataset: 2000 rows, `cueType` test accuracy 0.898
- numeric outputs are still baseline regressions and should be improved after feedback-derived labels are collected

Use `AUI_TRAINING_REPORT.md` for the latest training metrics and interpretation.

## Next Dataset Target

Build an evaluation-derived dataset with this flow:

```text
1. Generate situations in Unity:
   distance, direction, view state, approach speed, speaking, crossing.
2. For each situation, present candidate cues:
   footstep, voice, ambient presence, clothing rustle, breathing, none.
3. Record objective and subjective measures:
   localization accuracy, reaction time, approach recognition, clarity, naturalness, discomfort.
4. Select the best cue and compute score targets.
5. Train and evaluate the cue-control model on unseen situations.
```
