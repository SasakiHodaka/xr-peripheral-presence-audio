# Research Story Format

この文書は、修士論文、研究発表、スライド作成で使う基本フォーマットをまとめたものである。

今後は、Scene Tokenそのものを最初に説明するのではなく、次の順序で研究の必要性を説明する。

```text
全体システム
  -> ユーザーストーリー
  -> 現状システムの課題
  -> 今回取り組む課題
  -> 提案手法
  -> 評価
```

目的は、「Scene Tokenを作った」という説明ではなく、「VR工場での協調作業支援システムの中で、なぜScene Tokenが必要なのか」を伝えることである。

## 1. 全体システム

最初に、研究全体の対象が何かを1枚で示す。

対象システムは、工場で複数人がVR上で協調作業するためのシステムである。

```text
VR Factory Collaboration System
|
|-- 作業者A
|-- 作業者B
|-- 作業者C
|
|-- 音声通信
|-- 空間音響
|-- 作業共有
`-- 状態共有
```

この段階では、Scene Tokenの詳細には入らない。まず、研究が扱う大きなシステムを示す。

研究全体の流れは次のように説明する。

```text
工場VR協調作業システム
  -> 作業者・環境状態の取得
  -> Scene Understanding
  -> [今回の研究] Scene Token生成
  -> 通信
  -> 空間音響レンダリング
  -> ユーザ
```

ここで重要なのは、「今回の研究はシステム全体のどこを改善するのか」を明確にすることである。

## 2. ユーザーストーリー

研究の必要性は、利用者目線で説明する。

```text
As a
VR工場で組立作業を行う作業員

I want
必要な情報だけを空間音声で受け取りたい。

So that
作業を止めることなく、周囲の状況を把握しながら安全かつ効率的に協調作業を進められる。
```

### 例1: 作業者からの呼びかけ

```text
作業者B: 「部品を持ってきて」
  -> 作業者Aは誰が話したか知りたい
  -> 右後ろから聞こえる
  -> すぐに反応できる
```

### 例2: 危険通知

```text
危険エリアで作業中
  -> 「フォークリフト接近」
  -> 重要な音として目立つ
  -> 事故を防げる
```

ユーザーストーリーでは、利用者が何に困り、何を受け取りたいのかを具体的に示す。

## 3. 現状システムの課題

現在のVR音声通信では、すべての音声をそのまま送る構成になりやすい。

主な課題は次の通りである。

- 全ての音声を送信している
- 音の重要度を考慮していない
- 必要な情報と不要な情報を区別できない
- 通信量が大きい
- 重要な発話が埋もれる

その結果、次の問題が起きる。

- 通信負荷が高い
- 状況把握しづらい
- 重要な発話に気づきにくい
- 作業効率が低下する
- 安全性の低下につながる可能性がある

ここでは、単に「通信量が多い」と説明するのではなく、「だから作業員が困る」という利用場面までつなげる。

## 4. 今回取り組む課題

本研究では、次の課題に着目する。

```text
VR工場での協調作業において、必要な情報だけを表現・伝送するための情報表現が存在しない。
```

現在の構成では、音声波形をそのまま送る。

```text
音声波形
  -> 全部送る
  -> 通信量が大きい
  -> 重要な情報と不要な情報を区別しにくい
```

提案する構成では、必要な情報をScene Tokenとして表現する。

```text
Scene Token
  -> 話者
  -> 方向
  -> 距離
  -> 発話状態
  -> 重要度
  -> イベント
```

この研究の焦点は、VR工場協調作業支援システム全体を完成させることではない。全体システムの中で、「通信量削減と状況把握の両立」に必要な情報表現を提案・評価することである。

## 5. 提案手法

ここで初めてScene Tokenを提案手法として説明する。

```text
Scene Understanding
  -> Scene Token生成
  -> 通信
  -> 空間音響レンダリング
```

Scene Tokenは、VR空間内の発話者や状況を、空間音響レンダリングに利用できる構造化情報として表現する。

例:

```json
{
  "speakerId": "B",
  "direction": "BACK_RIGHT",
  "distance": "MID",
  "speakingState": "SPEAKING",
  "turnState": "TURN_HOLDER",
  "semanticToken": "INSTRUCTION",
  "importance": "HIGH"
}
```

この表現により、音声波形をすべて同じように扱うのではなく、「誰が、どこから、どのような意味・重要度で発話しているか」を利用して提示を制御できる。

## 6. 評価

評価では、Scene Tokenによって通信量削減と状況把握支援が両立できるかを確認する。

候補となる評価項目は次の通りである。

- 通信量
- 話者定位精度
- 話者識別時間
- 状況理解精度
- 重要発話への気づきやすさ
- 作業効率
- 主観評価

発表では、提案手法の説明だけで終わらず、「Scene Tokenを使うことで、ユーザーストーリー上の困りごとがどの程度改善されたか」を評価に戻して説明する。

## 発表・論文での基本構成

今後の発表や論文では、次の構成を基本にする。

```text
1. 全体システム
   VR工場での協調作業支援システムを示す

2. ユーザーストーリー
   作業員が何をしたいのか、なぜ必要なのかを示す

3. 現状の課題
   全音声送信、重要度未考慮、通信負荷、状況把握困難を示す

4. 今回取り組む課題
   必要な情報だけを表現・伝送する情報表現の不足に絞る

5. 提案手法
   Scene Tokenの定義、生成、通信、レンダリングを説明する

6. 評価
   通信量、状況理解、作業効率、主観評価で検証する
```

## 重要な位置付け

Scene Tokenを主役にしすぎない。

本来の研究の位置付けは次の通りである。

```text
VR工場での協調作業支援システムという大きなシステムがある。
その中で、通信量削減と状況把握の両立という課題がある。
その課題を解決するために、必要な情報だけを表現・伝送するScene Tokenを提案する。
```

この順序で説明すると、「何を作ったか」ではなく、「なぜ必要で、システム全体のどこを改善する研究なのか」が伝わりやすくなる。
